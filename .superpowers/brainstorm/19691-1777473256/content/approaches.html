Architecture Approaches

How to add a backend to your static Next.js site

A

Static Export + Supabase Client-Side ★ Recommended

Keep output: "export" and the existing Netlify deployment exactly as-is. Add Supabase's JS SDK — your app talks directly to Supabase from the browser. Supabase's Row Level Security (RLS) policies enforce who can read/write what. No deployment changes, no new infrastructure.

Pros

  • Zero deployment changes
  • Proven pattern for booking apps
  • Supabase auth built-in (email + password)
  • RLS is genuinely secure
  • Fastest to ship

Cons

  • All logic lives in the browser
  • Complex RLS policies to write
B

Switch to SSR on Netlify + Supabase Server-Side

Remove output: "export" and enable Next.js server-side rendering. Add API routes that handle all mutations (book, cancel, manage sessions) with server-side Supabase. The existing marketing pages still work — they just render server-side instead of statically.

Pros

  • Best security model
  • Server-side validation
  • More flexible long-term

Cons

  • Requires deployment changes
  • Risk of breaking existing pages
  • More complex setup
  • Slower to ship
C

Static Export + Netlify Functions

Keep static export, but add Netlify serverless functions as a lightweight API layer. Functions call Supabase server-side and expose endpoints your app calls. A hybrid between A and B.

Pros

  • Keeps static export
  • Some server-side logic

Cons

  • Most complex to set up
  • Two separate code paths to maintain
  • Functions are slower cold-start than Supabase direct
